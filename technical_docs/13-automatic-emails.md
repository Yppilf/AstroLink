# Email Delivery System

## Overview

The Email Delivery System is responsible for sending all automated transactional emails generated by the platform.

Examples include:

* Application notifications
* Status updates
* Document signing requests
* Account-related communication
* Administrative notifications

The system integrates with Brevo and uses Brevo-hosted HTML templates combined with dynamically supplied data.

This approach separates email presentation from application logic, allowing email content and branding to be updated without modifying platform code.

---

# Architecture

The email subsystem consists of three layers:

```text
Application Event
        ↓
send_dynamic_email()
        ↓
Brevo API
        ↓
Brevo Template Rendering
        ↓
Recipient Inbox
```

The platform does not generate HTML email content directly.

Instead:

1. The application selects a custom-configured Brevo template.
2. Dynamic data is supplied.
3. Brevo renders the final email.
4. Brevo delivers the message.

---

# Brevo Integration

## API Authentication

Communication with Brevo is authenticated using an API key:

```python
BREVO_API_KEY
```

The key is loaded from environment variables:

```python
BREVO_API_KEY = os.getenv("BREVO_API_KEY")
```

This key should never be committed to source control and must be configured through deployment environment settings.

---

## Transactional Email API

Emails are sent using Brevo's Transactional Email API.

The integration uses:

```python
sib_api_v3_sdk.TransactionalEmailsApi
```

which provides direct access to Brevo's transactional email infrastructure.

---

# Email Templates

## Template Storage

Email templates are not stored inside the application.

Instead they are maintained directly within the Brevo platform.

Advantages include:

* Centralized email management
* Non-developer editing
* Consistent branding
* Reduced application complexity
* Template versioning within Brevo

---

## Template Selection

Each email is identified using a Brevo template identifier:

```python
template_id
```

Example:

```python
send_dynamic_email(
    recipients,
    template_id=12,
    dynamic_data=data
)
```

The application only needs to know the numeric template ID.

---

## Dynamic Data Injection

Brevo templates can contain placeholders.

Example template:

```html
Hello {{ params.name }},

Your application "{{ params.project }}" has been approved.
```

The application supplies matching data:

```python
{
    "name": "John Doe",
    "project": "Machine Learning Research"
}
```

Brevo automatically substitutes values when rendering the email.

# Sending Emails

## Core Function

All transactional emails are sent through:

```python
send_dynamic_email(
    recipients,
    template_id,
    dynamic_data
)
```

This acts as the central entry point for email delivery.

---

## Recipient Handling

Recipients can be provided as:

### Single Recipient

```python
"user@example.com"
```

### Multiple Recipients

```python
[
    "user1@example.com",
    "user2@example.com"
]
```

Single email addresses are automatically converted into lists.

---

## Sender Configuration

All emails are sent from:

```text
noreply@siriusa.nl
```

with display name:

```text
Sirius A
```

Reply messages are directed to:

```text
astrolink@siriusa.nl
```

This separation allows automated delivery while still providing a monitored contact address.

---

# Email Workflow

## Step 1: Trigger

An application event occurs.

Examples:

* New application submitted
* Application status changed
* Signature request generated
* Document completed

The event calls:

```python
send_dynamic_email(...)
```

---

## Step 2: Build Request

The system constructs a Brevo payload:

```python
SendSmtpEmail(
    to=[...],
    sender={...},
    reply_to={...},
    template_id=template_id,
    params=dynamic_data
)
```

This payload contains:

* Recipients
* Sender information
* Template selection
* Dynamic template variables

---

## Step 3: Submit to Brevo

The payload is sent through:

```python
api_instance.send_transac_email(...)
```

Brevo then takes responsibility for:

* Template rendering
* Queueing
* Delivery
* Bounce handling
* Reputation management

---

## Step 4: Logging

The result is recorded locally.

Successful sends are written to:

```text
email_attempts.csv
```

including:

* Date
* Recipients
* Template ID
* Dynamic data
* Delivery status
* Brevo message identifier

---

# Delivery Logging

## Email Attempt Log

Every send attempt is recorded.

File:

```text
email_attempts.csv
```

Tracked information:

| Field        | Description                  |
| ------------ | ---------------------------- |
| Date         | Send attempt date            |
| Recipients   | Target email addresses       |
| Template ID  | Brevo template used          |
| Dynamic Data | Parameters supplied          |
| Status       | Sent or Error                |
| Error        | Failure reason if applicable |

This provides a basic audit trail for outbound email activity.

---

## Example Entry

```csv
date,recipients,template_id,status,error
2025-04-01,user@example.com,12,sent,
```

---

# Failure Handling

## API Failures

Brevo may reject a request due to:

* Invalid template ID
* Invalid recipient address
* Account limitations
* Authentication problems
* Service outages

These errors raise:

```python
ApiException
```

which is caught and logged.

---

## Unexpected Errors

Any other exception is also caught and logged.

Examples:

* Serialization failures
* Missing configuration
* Runtime exceptions

The email subsystem is designed to fail gracefully without crashing the main application.

---

# Unsent Email Tracking

## Purpose

Failed emails are recorded for later investigation.

File:

```text
unsent_emails.csv
```

This ensures that lost notifications can be identified and potentially resent.

---

## Stored Information

| Field        | Description         |
| ------------ | ------------------- |
| Date         | Failure date        |
| Recipients   | Intended recipients |
| Template ID  | Target template     |
| Dynamic Data | Parameters supplied |
| Reason       | Failure cause       |

---

## Example Entry

```csv
date,recipients,template_id,reason
2025-04-01,user@example.com,12,Brevo API error
```

# Operational Monitoring

## Message IDs

Successful deliveries store the Brevo message identifier:

```python
response.message_id
```

This allows administrators to correlate:

* Application events
* Brevo delivery logs
* User-reported email issues

when troubleshooting delivery problems.

```text
Email triggered

```

messages to application output.

# Security Considerations

## API Key Protection

The Brevo API key grants permission to send email on behalf of the platform.

It must:

* Remain outside source control
* Be stored in environment variables
* Be restricted to deployment environments

---

## Data Exposure

Dynamic template parameters are transmitted to Brevo.

Therefore:

* Only necessary data should be included
* Sensitive information should be minimized
* Templates should avoid exposing confidential platform data

---

## Sender Reputation

Using Brevo centralizes:

* Bounce handling
* Spam protection
* DKIM signing
* SPF validation
* Delivery reputation

This improves overall deliverability compared to self-hosted email solutions.

---

# Design Rationale

The email subsystem was designed around several principles.

### Separation of Concerns

Application code determines:

* When an email is sent
* Which template is used
* What data is included

Brevo determines:

* How the email looks
* How it is delivered

---

### Maintainability

Template changes can be performed without application deployments.

Marketing, branding, and content updates remain independent from software releases.

---

### Reliability

Brevo provides a mature transactional email infrastructure with:

* High deliverability
* Queue management
* Bounce processing
* Delivery analytics

---

### Auditability

All send attempts and failures are recorded locally, creating an operational history that can be reviewed when diagnosing communication issues.

This architecture provides a robust and maintainable transactional email solution while keeping email content management separate from application logic.
